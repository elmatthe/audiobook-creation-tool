> **IMPORTANT — the original instruction file's theory was wrong.** The dropped plan
> (`0.3.2-mp3-speedup-bug.md`) assumed a **sample-rate mismatch** (22050 vs 44100). That was
> **disproven on the Mac with ffprobe**: the source M4B and the bad MP3 are **both 44100 Hz stereo**.
> The real cause is the source codec profile — **xHE-AAC (USAC)** — which ffmpeg's native `aac`
> decoder cannot decode (it drops ~24% of packets, so the re-encoded MP3 is shorter and plays
> ~1.3× too fast / choppy). The fix is therefore **decode-side only**, and only the **M4B Converter**
> (the one tool that ingests AAC) needed changing. `mp3_tool.py` and `m4b_maker.py` ingest MP3, not
> AAC, so they were **not** touched, contrary to the original plan's file list.

## How the fix was confirmed on the Mac (evidence)

- ffprobe — source `test-files/Reincarnated as a Sword.m4b`: `codec=aac profile=xHE-AAC
  sample_rate=44100 channels=2 duration=35199.6s`. Bad `…Sword.mp3`: `mp3 44100 2 duration=26783.4s`
  (ratio 0.761 → ~1.31× fast). **Sample rates match → not a rate bug.**
- Native `aac` decoder emits `Error submitting packet to decoder: Not yet implemented in FFmpeg,
  patches welcome` thousands of times and drops packets.
- Reproduction: a **600 s** source slice through the converter's exact command produced **454.6 s**
  (native) → fixed to **600.0 s** with `-c:a aac_at`.
- A normal AAC-LC audiobook in `test-files/` decodes cleanly and is left untouched by the new logic.

## ⚠ Open item for the Windows host BEFORE cutting the release

`aac_at` (Apple AudioToolbox) is **macOS-only**. On Windows, `input_decoder_args()` returns `[]`
and the native decoder is used. **You must test on Windows whether the bundled ffmpeg decodes
xHE-AAC correctly**, using the same `test-files\Reincarnated as a Sword.m4b`:

1. If the Windows ffmpeg **decodes xHE-AAC correctly** (output duration ≈ source duration): the fix
   is complete cross-platform — nothing more to do. Proceed to release.
2. If the Windows ffmpeg **also mis-decodes it**: the new **duration guard will fail the conversion
   with a clear message** (no silent sped-up file), but the user still cannot convert that file on
   Windows. Decide before release whether to (a) ship as-is (macOS-fixed; Windows fails loudly), or
   (b) source a Windows ffmpeg build that decodes USAC/xHE-AAC and bundle it into `resources/bin/`.
   This is a **decision for the maintainer** — do not invent a Windows decoder silently.

Quick Windows probe + repro (PowerShell / cmd):
```bat
ffprobe -v error -select_streams a:0 -show_entries stream=profile,sample_rate,channels,duration ^
  -of default=noprint_wrappers=1 "test-files\Reincarnated as a Sword.m4b"
:: convert a 600s slice with the bundled ffmpeg's native decoder and check the output duration
ffmpeg -hide_banner -y -ss 0 -t 600 -i "test-files\Reincarnated as a Sword.m4b" -vn ^
  -c:a libmp3lame -q:a 2 "%TEMP%\win_probe.mp3"
ffprobe -v error -show_entries format=duration -of default=nk=1:nw=1 "%TEMP%\win_probe.mp3"
:: ~600  -> Windows decodes xHE-AAC fine (case 1).  ~455 -> mis-decodes (case 2).
```
